Comment Procéder

Cet exercice couvre une majorité des éléments vus en TP, dans un seul script à completer, qui sera fourni (script.sh). Il sera décomposé en questions, chacune étant indépendante des autres.

Une rapide description de ce qui est demandé est donnée dans le sujet pour chaque question, et une section correspondante à compléter dans script.sh.

Question démo

Nous souhaitons récupérer une entrée utilisateur, qui sera stockée dans la variable coucou. La partie à compléter pour chaque question sera remplie avec des valeurs par défaut hardcodées. 

# Question démo
echo "Question démo"
coucou=5
echo "coucou =" $coucou

Certains éléments étant plus difficiles que d'autres, il vous est possible de ne pas les traiter, en laissant telles quelles les déclarations de variables hardcodées. Cela vous permettra de traiter la suite des questions sans soucis.

1.2 Question usage

Afin qu'un utilisateur lambda puisse utiliser notre script, il faut lui indiquer comment le faire, c'est-à-dire quels arguments sont nécessaires et quelles sont les options disponibles.

Pour cela, vous allez réaliser une fonction usage(), affichant un message du type : 

Usage: ./script.sh [options]
-h : afficher l'aide

1.3 Question argument(s)

Avant d'effectuer le traitement de notre commande à proprement parler, il faut récupérer le(s) argument(s) passé(s) en paramètre par l'utilisateur lors de l'appel de la commande.

Ici, nous voulons récupérer le chemin du fichier de configuration, et vérifier qu'il correspond bien à un fichier, puis le stocker dans une variable config_file. Dans le cas contraire, il faudra appeler la fonction usage définie précédemment et quitter le programme avec le bon code. (Dans l'éventualité ou vous n'auriez pas fait la fonction usage, affichez simplement un texte pertinent.)

NB : N'oubliez pas de faire un shift après avoir stocké la valeur de l'argument, autrement la partie de traitement des options avec getopts ne fonctionnera pas.

1.4 Question option(s)

Afin de rendre notre commande à la fois utilisable simplement, et paramétrable plus finement (par un utilisateur averti, par exemple), nous utiliserons des options.

Dans notre cas, nous souhaitons ajouter deux options :

-h
Notre commande affichera l'usage (via l'appel de usage si définie précédemment) et quittera avec le bon code de retour (cf 1).
-v 2
Cette option permet de définir un niveau de verbosité de notre commande. Autrement dit, à quel point le texte affiché doit être détaillé.

Dans le cas où une option non prise en charge est passée par l'utilisateur, on doit afficher l'usage et quitter avec le code de retour correspondant.

Vous utiliserez pour cela la commande build-in getopts. 

# Question option(s)
verbosity=1

1.6 Question config

Parfois, de nombreux paramètres sont nécessaires pour configurer un programme selon les besoins spécifiques de l'utilisateur. Il n'est alors pas pratique d'appeler une commande avec de nombreuses options.

Une solution commune consiste à lire un fichier de configuration contenant ces informations.

Voici un exemple de fichier de configuration, dont nous allons nous servir pour ce TP. 

[User parameters]
coucou1 = 25
faux_param_de_remplissage = "75 truc"

[Python]
CLEAN_PYTHON = .env|.pyg|pycache

[Latex]
CLEAN_LATEX = .aux|.bbl|.bcf|.log|.fls|.toc|.cb2|.synctex.gz|.gls|.brf

Ici, on peut voir que plusieurs informations sont disponibles, dont des règles de nettoyage de projets en fonction du langage de ce projet.

L'objectif de cette question est de récupérer un paramètre spécifique, la liste des extensions de fichiers à supprimer pour nettoyer un projet Latex de ses fichiers intermédiaires de compilation qui ne nous intéressent pas.

Pour ce faire, nous utiliserons awk avec une expression régulière pour trouver la bonne ligne, ainsi qu'un split (dans awk) pour séparer correctement les valeurs du paramètre CLEAN_LATEX dans un tableau, qui sera ensuite parcouru (toujours dans awk) afin d'afficher ses valeurs.

Cet affichage n'est cependant pas suffisant, il nous faut récupérer la chaîne de caractères ainsi créée pour l'utiliser plus tard.

Il suffit alors d'assigner la valeur de retour d'awk à une variable avec la syntaxe suivante : 

to_clean=$(awk ...........)

Cette question est difficile, si vous n'avez pas déjà une solide idée de comment faire, il serait judicieux de passer à la suite (laissez alors la variable hardcodée). 

# Question config
to_clean=".aux .log .fls .toc .cb2 .synctex.gz .gls .brf"
[[ $verbosity -ge 2 ]] && echo "Les fichiers portants ces extensions vont êtres supprimés :"
[[ $verbosity -ge 1 ]] && echo "TO_CLEAN=${to_clean}"

Vous pouvez également voir ici comment on utilise verbosity, pour afficher plus de texte si la valeur est plus grande.

1.7 Question nettoyage

Maintenant que nous savons quels types de fichier nettoyer, il faut parcourir les fichiers du dossier courant, et tester si son extension est dans la chaîne to_clean.

Pour chaque fichier, il faudra :

extraire son extension
En effet, seule l'extension est pertinente pour décider de la suppression ou non du fichier. Pour rappel, l'extension d'un fichier:
~/mes_projets/coucou.pdf

Est uniquement : 

pdf
tester l'extension
On cherche ici à déterminer si oui ou non l'extension est présente dans la chaîne to_clean. L'astuce consiste à créer une expression régulière à partir de ext, et de voir si to_clean matche cette expression. Afin d'intégrer une variable dans l'expression régulière, il faut l'encapsuler dans des double quote. Voici un exemple :
ext="pdf"
regex="truc${ext}bidule"
echo $regex

trucpdfbidule
Supprimer le fichier
Uniquement dans le cas où cela matche, alors on affiche un texte explicatif en fonction de verbosity.

Il vous est donc demandé de compléter la partie ci-dessous dans votre script. 

# Question nettoyage
for f ......
do
  ext=.....
  regex="......"
  if [[ $to_clean =~ $regex ]]
  then
      [[ $verbosity -ge 2 ]] && echo "Le fichier ${f} va être supprimé."
      [[ $verbosity -ge 1 ]] && echo "Appel de : rm ${f}"
      rm $f
  fi
done
